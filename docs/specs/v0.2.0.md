# Muninn Design Notes: Schema References & Backlinks

## North Star (anchor for all decisions below)

Muninn is the PKM that **gets out of your way** and **makes using a PKM actually valid**. Every feature should either:

- Make existing notes more useful without creating maintenance work, or
- Reduce the tax of capturing notes correctly in the first place.

Features that create meta-notes-management (graph views, "related notes" suggestions, dashboards-of-dashboards) fail
this test and are out of scope — even when they look reasonable.

The sharper version of the test: Muninn is the inverse of Obsidian. The question isn't "would Obsidian users like
this?" — it's "does this help a 10K-note vault stay manageable, or does it add a meta-surface to maintain?"
Obsidian-shaped feature accumulation is the failure mode to actively avoid, not just one to passively decline.

## The two-layer graph model

Muninn supports two distinct mechanisms for expressing relationships between notes. They aren't competing — they address
different needs and coexist cleanly.

### Layer 1: Typed metadata references (primary, schema-driven)

Schemas can declare frontmatter fields as references to other notes:

```yaml
- key: instructor
  type: reference
  target: 'people.**'
  required: true
```

When a field has `type: reference`:

- The value is a **canonical dot-path** (e.g. `people.john-smith`), not a display name. This keeps references stable
  across renames and unambiguous when display names collide.
- Validation verifies the target exists and matches the `target` pattern.
- Refactor updates these fields atomically when a referenced note is renamed.
- The backlink index records _which field_ on _which note_ referenced the target — so backlinks can be rendered grouped
  by relationship type ("Referenced as instructor by: …", "Referenced as attendee by: …").

This is the right primary mechanism because:

- The reference is a **byproduct** of writing a correct, schema-validated note. The user already had to fill in
  `instructor`. The graph edge is free.
- Edges are **typed** — far more useful than untyped wikilinks for queries like "every training where John was
  instructor of record" (the EHS compliance question, not "every note that mentions John").
- Schemas elevate from "validate fields" to "declare the graph structure of the vault." Coheres with the existing schema
  philosophy.

### Layer 2: Wikilinks (secondary, prose-level)

`[[wikilink]]` syntax for cross-references that don't fit a schema — the casual mention of `[[acme-contract]]` in a
meeting note, the prose-level "see also" links.

- Always works, no schema required.
- Backlinks-at-the-bottom render these in a separate section ("Mentioned by: …") so they're visually distinct from typed
  references.
- This is the layer Obsidian-migrants will recognize and lean on. They never have to engage with the schema-reference
  layer if they don't want to.

### Why the dual model is the right design

- **Obsidian-style users** can ignore schemas entirely. They write notes, write wikilinks, get backlinks. The
  metadata-reference machinery is invisible to them.
- **Schema-driven users** (EHS workflow, etc.) get rich typed edges with compliance-grade query power, and still have
  wikilinks available for ad-hoc prose references.
- Neither group has to know the other group's mode exists. The tool meets both where they are.

**Critical: schemas must remain opt-in for typed references.** Don't punish wikilink-only users by gating backlinks
behind schema declaration. Users choose how much structure they want; the tool shouldn't impose it.

## Backlinks-at-the-bottom: rendering & storage

Foam-style: resolved backlinks rendered into the document at the bottom. Both reference types feed the same index, but
render distinctly.

### On-disk vs. virtual rendering

**Decision: on-disk, in a fenced auto-managed section.**

```markdown
<!-- muninn:backlinks -->

... auto-generated content ...

<!-- /muninn:backlinks -->
```

Rationale:

- **On-disk** keeps notes valid when Muninn isn't running (cat, GitHub preview, grep all still work). Aligns with the
  "plain markdown on disk" principle.
- **Fenced markers** give a clean regeneration target — replace what's between markers, leave the rest of the document
  untouched. No need to re-parse the whole note.
- Diff noise is the trade-off, but it's bounded (only when links change) and arguably useful as provenance.

### What counts as a backlink (initially)

- Explicit `[[wikilinks]]` → wikilink-style backlinks.
- Schema-declared reference fields → typed backlinks, grouped by field.

### Index implementation

At ~10k notes, full re-scan on every save is unworkable. The pattern:

- In-memory inverted index in the Go sidecar: `note → [(referencing_note, reference_kind, field_name?)]`
- Updated incrementally on file change events (`fsnotify`).
- Snapshot persisted to disk on shutdown for fast cold start.
- Same underlying link-graph data serves lookup and backlinks — build once, query two ways.

## v0.2+ sequencing implications

The dual-layer graph model changes the right ordering of upcoming work, because the architectural foundations need to
land before the user-visible features can ship coherently.

**v0.2 — Refactor, schema-aware from day one**

The renaming engine knows about `type: reference` fields and updates them atomically alongside link/filename updates.
Even if reference fields aren't yet documented for users, refactor is built to handle them.

**v0.2 or v0.3 — Schema `type: reference` field type**

Validation, target-pattern matching, missing-target errors. No backlinks UI yet — this just makes schemas capable of
declaring graph structure. Dogfood-able in EHS schemas before any UI ships.

**v0.3 or v0.4 — Backlinks-at-the-bottom (both kinds)**

By the time this ships, users who've been declaring reference fields already have correct graph data and get rich typed
backlinks immediately. Wikilink users get the simpler version. Neither group is surprised.

**Independent, slot in whenever — Templates**

Templates use VS Code's existing snippet syntax with a small registered variable surface (`today`, `user`,
`frontmatter`, `hierarchy`). Revisit only if real use cases demand computation snippets can't express. JS templates
are explicitly _not_ the plan — no use case has surfaced that requires them, and the sandbox + porting-guide
maintenance tail is exactly the Obsidian-shaped accumulation the North Star rejects.

### Why this order matters

If wikilink backlinks ship first and metadata references arrive later, users will have written wikilinks for
relationships that _should_ have been schema fields (because that was the only mechanism available). When typed
references arrive, those notes feel "wrong" and there's pressure to migrate or accept inconsistency.

Landing the foundation first means the structured option is available from early on, the underlying graph data is
correct from the start, and the UI ships on top of solid ground.

## Open questions to revisit after real usage

- Does requiring canonical dot-paths in YAML reference values feel annoying enough in practice to warrant a quick-pick
  UI for filling them in?
- How does refactor handle the case where a reference target is itself being renamed mid-refactor? (Probably: resolve
  the rename graph topologically, update in dependency order.)
- Do `{{today}}` / `{{user}}` template magics survive the arrival of JS templates, or are they redundant once a sandbox
  can compute them? (Taste call — leaning keep, since they're convenient for non-JS users.)
- Does the hierarchy get _looser_ once backlinks exist? (Likely yes — filing decisions matter less when cross-cutting
  relationships stay visible. Watch for it; may relax some schemas.)

## Anti-features to keep saying no to

These will be tempting. They are out of scope by North Star, not by oversight.

- Graph view (creates a meta-surface to maintain a mental model of)
- Daily-note dashboards (notes-about-notes)
- Related-notes suggestions / recommendation engines (tuning + disagreement tax)
- Backlinks panel as a separate UI surface (defeats the "in context only" property of backlinks-at-the-bottom)
- Frontmatter-as-backlinks for _arbitrary_ string fields (only schema- declared reference fields count — anything else
  is magic)
- Drop-in compatibility with Dendron/Foam template execution semantics (port, don't shim)

The "that is it" energy is the most valuable thing in the project. When tempted, re-read this section.
